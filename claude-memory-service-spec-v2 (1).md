## Overview

A serverless AWS microservice that provides persistent, semantically-searchable memory storage for Claude across all conversations and projects. Claude connects to the service via a local MCP server using mutual TLS (mTLS) authentication. Both the server and client certificates are issued by ACM as exportable public certificates — AWS manages issuance and renewal, and the private key can be exported for use by the MCP server client. Memory vectors are generated by Amazon Bedrock Titan Text Embeddings v2 and stored in S3 Vector Tables with dual-scope namespacing (per-project and global).

## Certificate Architecture

This service uses ACM Exportable Public Certificates, launched by AWS in June 2025\. These are ACM-issued public certificates from Amazon Trust Services where the private key is accessible via the `ExportCertificate` API — enabling mTLS client authentication without operating a private CA.

### Certificate inventory

| Cert | Type | Purpose | Cost |
| :---- | :---- | :---- | :---- |
| `memory.brad-duhon.com` | ACM public (non-exportable) | API Gateway custom domain server TLS | Free |
| `mcp-client.brad-duhon.com` | ACM exportable public | MCP server client cert for mTLS | $15 one-time |

### How ACM exportable certs work

ACM (Amazon Trust Services)

    │

    ├── issues server cert  →  attached to API Gateway custom domain ARN (AWS-managed)

    │                          standard ACM cert, auto-renewed, free

    │

    └── issues exportable client cert  →  aws acm export-certificate → encrypted PEM bundle

                                          private key stored in Secrets Manager

                                          MCP server fetches key on startup

### Truststore design

API Gateway's mTLS truststore must contain the CA certificate(s) that signed the client cert. Because the client cert is issued by Amazon Trust Services (ACM's public CA), the truststore is the **Amazon Trust Services root CA bundle** — a publicly available PEM file. This is uploaded to S3 and referenced by API Gateway's mTLS configuration.

Truststore S3 object: s3://brad-claude-memory-{account}/mtls/truststore.pem

Contents: Amazon Trust Services Root CA bundle (public, no secrets)

**Note:** The truststore contains only public CA certificates — no private keys, no sensitive material. It can be version-controlled and treated as configuration.

### Client cert export flow

\# Export the client cert private key from ACM (run once, store result in Secrets Manager)

aws acm export-certificate \\

  \--certificate-arn arn:aws:acm:us-east-1:{account}:certificate/{cert-id} \\

  \--passphrase fileb://passphrase.txt \\

  | jq \-r '"\\(.Certificate)\\(.CertificateChain)\\(.PrivateKey)"' \\

  \> client-bundle.pem

\# Store encrypted bundle in Secrets Manager

aws secretsmanager create-secret \\

  \--name memory-service/mcp-client-cert \\

  \--secret-string file://client-bundle.pem

The MCP server fetches the cert bundle from Secrets Manager on startup and decrypts it using the passphrase (stored as a separate secret). The private key never touches disk on any server — it lives in Secrets Manager and in-process memory on the MCP host.

### Cert renewal automation

ACM automatically renews exportable public certs before expiry. When renewal occurs, the new cert must be re-exported and the updated bundle pushed to Secrets Manager. An EventBridge rule on the `ACM Certificate Approaching Expiration` event triggers a Lambda that:

1. Calls `acm:ExportCertificate` with the stored passphrase  
2. Overwrites the Secrets Manager secret with the new bundle  
3. Publishes an SNS notification for awareness

EventBridge rule: source \= "aws.acm", detail-type \= "ACM Certificate Approaching Expiration"

Target: cert-rotation Lambda

### API Gateway

| Property | Value |
| :---- | :---- |
| Type | HTTP API (not REST API — lower cost, sufficient features) |
| Server TLS | Standard ACM public cert on custom domain — free, auto-renewed |
| mTLS auth | Enabled — truststore S3 URI: `s3://brad-claude-memory-{account}/mtls/truststore.pem` |
| Custom domain | Required for mTLS (`memory.brad-duhon.com`) |
| Routes | `POST /store`, `POST /recall`, `POST /summarize` |
| Integration | Lambda proxy integration |
| Throttling | 100 RPS burst, 50 RPS steady (single-user service) |
| Client identity header | `x-amzn-mtls-clientcert-subject` forwarded to Lambda |

**Note:** API Gateway validates the client cert chain against the truststore (Amazon Trust Services root) before the request reaches Lambda. Lambda asserts the forwarded `x-amzn-mtls-clientcert-subject` header matches the expected MCP client CN as a defense-in-depth check.

### Lambda Function

| Property | Value |
| :---- | :---- |
| Runtime | Python 3.12 or Node.js 20 |
| Architecture | arm64 (Graviton2 — \~20% cheaper, faster cold starts) |
| Memory | 512 MB (sufficient for embedding \+ S3 ops) |
| Timeout | 30s (recall \+ summarize may take 10–15s) |

| Concurrency | Reserved: 10 (single-user; prevents runaway invocations) |
| Environment vars | `MEMORY_BUCKET`, `BEDROCK_REGION`, `EMBED_MODEL_ID`, `EXPECTED_CLIENT_CN` |
| Layers | AWS SDK (built-in), boto3 (built-in for Python) |

**Handler dispatch logic:**

def handler(event, context):

    tool \= event\["tool\_name"\]

    client\_subject \= event\["headers"\].get("x-amzn-mtls-clientcert-subject")

    assert\_trusted\_client(client\_subject)  \# defense-in-depth CN assertion

    match tool:

        case "store\_memory":        return handle\_store(event)

        case "recall\_memory":       return handle\_recall(event)

        case "summarize\_memories":  return handle\_summarize(event)

        case \_: return {"statusCode": 400, "body": "Unknown tool"}

def assert\_trusted\_client(subject: str):

    expected\_cn \= os.environ\["EXPECTED\_CLIENT\_CN"\]  \# e.g. "mcp-client.brad-duhon.com"

    if expected\_cn not in (subject or ""):

        raise PermissionError(f"Untrusted client subject: {subject}")

### MCP Server

| Property | Value |
| :---- | :---- |
| Runtime | Node.js 20 (MCP SDK) or Python (mcp library) |
| Location | Local process — runs on your workstation or dev host |
| Auth | Client cert \+ private key fetched from Secrets Manager on startup |
| Key storage | AWS Secrets Manager — fetched at runtime, held in-process memory only |
| Config | `~/.config/claude/mcp_servers.json` (Claude Code) or claude.ai MCP settings |
| Exposed tools | `store_memory`, `recall_memory`, `summarize_memories` |

**Startup cert fetch pattern:**

// On MCP server startup — fetch cert bundle from Secrets Manager

const { SecretsManagerClient, GetSecretValueCommand } \= require("@aws-sdk/client-secrets-manager");

async function loadClientCert() {

  const client \= new SecretsManagerClient({ region: "us-east-1" });

  const bundle \= await client.send(new GetSecretValueCommand({

    SecretId: "memory-service/mcp-client-cert"

  }));

  const passphrase \= await client.send(new GetSecretValueCommand({

    SecretId: "memory-service/mcp-client-cert-passphrase"

  }));

  return { bundle: bundle.SecretString, passphrase: passphrase.SecretString };

}

**MCP server config entry:**

{

  "memory-service": {

    "command": "node",

    "args": \["/path/to/memory-mcp/index.js"\],

    "env": {

      "MEMORY\_API\_URL": "https://memory.brad-duhon.com",

      "AWS\_REGION": "us-east-1",

      "CERT\_SECRET\_ARN": "arn:aws:secretsmanager:us-east-1:{account}:secret:memory-service/mcp-client-cert",

      "PASSPHRASE\_SECRET\_ARN": "arn:aws:secretsmanager:us-east-1:{account}:secret:memory-service/mcp-client-cert-passphrase"

    }

  }

}

**Security note:** The private key is never written to disk on the MCP host. It is fetched from Secrets Manager into process memory at startup and used directly by the HTTPS client for the mTLS handshake. If the process is restarted, it re-fetches from Secrets Manager — ensuring the key always reflects the current cert after rotation.

## Memory Namespace Design

All memory objects live under one of two key prefixes in the S3 Vector Table bucket:

s3://brad-claude-memory-{account}/

├── project/

│   └── {project\_id}/

│       └── memories/

│           ├── {uuid}.json          ← individual memory

│           └── summary-{uuid}.json  ← compressed summary

└── global/

    └── memories/

        ├── {uuid}.json

        └── summary-{uuid}.json

**Scoping rules:**

| Situation | Namespace written to |
| :---- | :---- |
| Conversation inside a Claude Project | `project/{project_id}/memories/` |
| Conversation outside any project | `global/memories/` |
| Explicit `store_memory` with `scope: global` | `global/memories/` always |
| `recall_memory` with `project_id` | Queries both prefixes; merges and ranks results |
| `recall_memory` without `project_id` | Queries `global/memories/` only |

## Memory Trigger Strategy

This service uses a **hybrid trigger model** combining automatic session-end summarization, explicit mid-conversation storage, daily compression, and a Claude Code `PostCompact` hook.

### Trigger 1 — Session end (automatic)

At the end of every conversation, Claude calls `store_memory` with a structured summary of the session. Implemented via system prompt instruction:

At the end of every conversation, before your final response, call store\_memory

with a concise summary covering: decisions made, preferences expressed,

technical context established, and any action items. Use scope: project if

inside a Claude Project, otherwise scope: global.

**What gets stored:** One embedding per conversation — 1 Bedrock call, 1 S3 write.  
**Signal quality:** High — session-level context, not message-level noise.

### Trigger 2 — Explicit mid-conversation (manual)

You instruct Claude at any point: *"remember this"*, *"store this decision"*, *"save that architecture choice"*. Claude calls `store_memory` immediately with the specific content indicated.

**What gets stored:** The exact content flagged — highly curated.  
**Signal quality:** Maximum — user-selected.

### Trigger 3 — Daily summarization (scheduled)

An EventBridge Scheduled Rule invokes the Lambda with `tool_name: summarize_memories` daily at 02:00 UTC. The summarizer:

1. Lists all memory objects created in the last 24 hours  
2. Groups by project prefix  
3. Sends batches to Claude Haiku via Bedrock with a compression prompt  
4. Writes a `summary` type object back to the same prefix  
5. Deletes the originals it compressed (configurable — retain initially)

### Trigger 4 — PostCompact hook (Claude Code only)

When Claude Code compacts the context window — either automatically (context limit approached) or manually (`/compact`) — a `PostCompact` shell hook fires and pushes the compaction summary directly to `store_memory`. This is the highest-signal trigger: it captures exactly what Claude determined was worth preserving before truncation, without relying on a session-end prompt or manual instruction.

See [Claude Code PostCompact Hook](#claude-code-postcompact-hook) for full implementation details.

**What gets stored:** The compaction summary output — Claude's own distillation of the conversation so far.  
**Signal quality:** Very high — Claude-generated under context pressure; captures the most load-bearing context.  
**Trigger field:** `compact_auto` or `compact_manual` — queryable in recall results.

### Trigger summary

| Trigger | Mechanism | Calls/month (est.) | Signal quality |
| :---- | :---- | :---- | :---- |
| Session end | System prompt instruction → MCP `store_memory` | 750 | High |
| Explicit | User instruction → MCP `store_memory` | 90 | Maximum |
| Daily summarize | EventBridge → Lambda `summarize_memories` | 30 runs | Compression pass |
| PostCompact hook | Claude Code hook → shell script → API | \~60 | Very high |

Compact hook estimate assumes compaction fires roughly 2–3× per week across 25 daily conversations — conservative given mixed session lengths.

### What NOT to do

Do not store every message. At 15–40 conversations/day with mixed length, storing every turn would generate 300–1,200+ embedding calls/day, produce a noisy vector store with low signal density, and degrade `recall_memory` result quality through cosine similarity dilution.

## Claude Code PostCompact Hook

The `PostCompact` hook fires in Claude Code immediately after context compaction completes — whether triggered automatically (context limit approached) or manually via `/compact`. At that moment, Claude Code provides the compacted summary as structured JSON on stdin to whatever shell command you configure. This hook pushes that summary directly to `store_memory` via an mTLS `curl` call, bypassing the MCP server process entirely.

### Why this is the best trigger

Compaction is the moment Claude has already produced a distilled, load-bearing summary of the conversation under real context pressure. It is Claude's own judgment about what matters — more authoritative than a session-end prompt heuristic, and available for long-running sessions that may never reach a clean end.

### Certificate placement for the hook script

The MCP server fetches its private key from Secrets Manager into process memory. The hook script is a plain shell command with no runtime AWS SDK — it needs the cert and key as files on disk. Export the decrypted key once during setup and place both files in `~/.claude/certs/`:

\# One-time setup — decrypt the key from Secrets Manager and place on disk

aws secretsmanager get-secret-value \\

  \--secret-id memory-service/mcp-client-cert \\

  \--query SecretString \--output text \> /tmp/bundle.pem

PASSPHRASE=$(aws secretsmanager get-secret-value \\

  \--secret-id memory-service/mcp-client-cert-passphrase \\

  \--query SecretString \--output text)

mkdir \-p \~/.claude/certs

\# Extract cert

openssl x509 \-in /tmp/bundle.pem \-out \~/.claude/certs/client.crt

\# Decrypt and extract private key

openssl pkey \-in /tmp/bundle.pem \-passin "pass:$PASSPHRASE" \\

  \-out \~/.claude/certs/client.key

\# Lock down permissions

chmod 700 \~/.claude/certs

chmod 600 \~/.claude/certs/client.crt \~/.claude/certs/client.key

\# Download Amazon Trust Services CA bundle for curl CA validation

curl \-s https://www.amazontrust.com/repository/AmazonRootCA1.pem \\

  \> \~/.claude/certs/amazon-trust-services-ca.pem

\# Clean up

rm /tmp/bundle.pem

unset PASSPHRASE

**Security note:** The private key file is stored at `chmod 600`, readable only by your user account. It never leaves your local machine. Re-run this setup after each annual ACM cert renewal — the EventBridge rotation Lambda updates Secrets Manager, and you pull the fresh bundle the same way.

### `~/.claude/settings.json` — hook configuration

This lives in the global user settings so it applies to every Claude Code session on your machine, not just a single project:

{

  "hooks": {

    "PostCompact": \[

      {

        "matcher": "auto|manual",

        "hooks": \[

          {

            "type": "command",

            "command": "\~/.claude/hooks/post-compact-memory.sh",

            "timeout": 15

          }

        \]

      }

    \]

  }

}

**Matcher values:**

- `auto` — fires when Claude Code compacts automatically on context limit  
- `manual` — fires when you run `/compact` explicitly  
- `auto|manual` — fires on both (recommended)

### `~/.claude/hooks/post-compact-memory.sh`

\#\!/bin/bash

set \-euo pipefail

\# Read PostCompact JSON from stdin

INPUT=$(cat)

\# Extract compacted summary and session metadata

SUMMARY=$(echo "$INPUT"    | jq \-r '.summary // empty')

SESSION\_ID=$(echo "$INPUT" | jq \-r '.session\_id // "unknown"')

PROJECT\_ID=$(echo "$INPUT" | jq \-r '.project\_id // empty')

TRIGGER=$(echo "$INPUT"    | jq \-r '.trigger // "auto"')   \# "auto" or "manual"

\# Nothing to store if summary is empty

if \[ \-z "$SUMMARY" \]; then

  exit 0

fi

\# Determine scope — project if we have a project\_id, otherwise global

if \[ \-n "$PROJECT\_ID" \] && \[ "$PROJECT\_ID" \!= "null" \]; then

  SCOPE="project"

else

  SCOPE="global"

  PROJECT\_ID="null"

fi

\# Build the store\_memory payload

PAYLOAD=$(jq \-n \\

  \--arg text          "$SUMMARY" \\

  \--arg scope         "$SCOPE" \\

  \--arg project\_id    "$PROJECT\_ID" \\

  \--arg conversation\_id "$SESSION\_ID" \\

  \--arg trigger       "compact\_$TRIGGER" \\

  '{

    tool\_name:       "store\_memory",

    text:            $text,

    scope:           $scope,

    project\_id:      (if $project\_id \== "null" then null else $project\_id end),

    conversation\_id: $conversation\_id,

    trigger:         $trigger

  }')

\# POST to memory service via mTLS

RESPONSE=$(curl \--silent \--show-error \\

  \--cert    "$HOME/.claude/certs/client.crt" \\

  \--key     "$HOME/.claude/certs/client.key" \\

  \--cacert  "$HOME/.claude/certs/amazon-trust-services-ca.pem" \\

  \--max-time 10 \\

  \-H "Content-Type: application/json" \\

  \-d "$PAYLOAD" \\

  "https://memory.brad-duhon.com/store")

\# Log result to stderr — appears in Claude Code verbose log, not Claude's context

STATUS=$(echo "$RESPONSE" | jq \-r '.stored // false')

if \[ "$STATUS" \= "true" \]; then

  echo "PostCompact memory stored (trigger: compact\_$TRIGGER, scope: $SCOPE)" \>&2

else

  echo "Warning: PostCompact memory store returned unexpected response: $RESPONSE" \>&2

fi

\# Always exit 0 — never block compaction on a memory write failure

exit 0

\# Make executable after creating

chmod \+x \~/.claude/hooks/post-compact-memory.sh

### How the hook behaves

| Scenario | Outcome |
| :---- | :---- |
| Memory API reachable, store succeeds | Memory written, logged to stderr, compaction continues |
| Memory API unreachable (timeout) | `curl` times out after 10s, warning logged to stderr, compaction continues |
| Empty summary from compaction | Script exits 0 immediately — nothing stored |
| Compaction inside a Claude Project | `scope: project`, `project_id` populated from session context |
| Compaction outside any project | `scope: global`, `project_id: null` |
| Annual cert renewal | Re-run the one-time setup block to pull fresh key from Secrets Manager |

### `trigger` field values in stored memories

| Trigger field | Source |
| :---- | :---- |
| `session_end` | System prompt → MCP `store_memory` at conversation close |
| `explicit` | User instruction → MCP `store_memory` mid-conversation |
| `compact_auto` | PostCompact hook, automatic compaction |
| `compact_manual` | PostCompact hook, `/compact` command |
| `summarizer` | Daily EventBridge → Lambda `summarize_memories` run |

## Security Controls

| Control | Implementation |
| :---- | :---- |
| Client authentication | mTLS — ACM exportable public cert presented by MCP server, validated at API Gateway |
| Truststore | Amazon Trust Services root CA bundle in S3 — public CA, no private material |
| Defense-in-depth auth | Lambda asserts `x-amzn-mtls-clientcert-subject` CN on every invocation |
| Transport encryption | TLS 1.2 minimum; TLS 1.3 preferred on API Gateway |
| Data at rest | S3 SSE-KMS |
| Network isolation | Lambda in VPC private subnet; S3 via Gateway Endpoint; Bedrock via Interface Endpoint |
| Bucket policy | Deny all S3 access not sourced from VPC endpoint |
| Private key at rest | Encrypted PEM bundle in Secrets Manager — never written to disk on any server |
| Private key in transit | Fetched from Secrets Manager into MCP server process memory at startup only |
| Cert renewal | ACM auto-renews; EventBridge triggers re-export to Secrets Manager automatically |
| Logging | All Lambda invocations logged to CloudWatch with tool name and latency |
| Concurrency cap | Reserved concurrency \= 10 — limits blast radius of runaway invocations |
| Versioning | S3 object versioning enabled — accidental deletes recoverable |

## Open Decisions

| Decision | Options | Recommendation |
| :---- | :---- | :---- |
| Lambda runtime | Python 3.12 vs Node.js 20 | Python 3.12 — boto3 is mature; MCP SDK has good Python support |
| Summarizer model | Claude Haiku (fast, cheap) vs Claude Sonnet (higher quality) | Haiku — sufficient for compression; revisit if summary quality is poor |
| Original retention post-summarize | Delete originals vs retain with `archived` tag | Retain with tag initially; delete after 30 days of confidence in summarizer |
| Recall top-K | Fixed at 5 vs configurable per-call | Start at 5; expose as optional parameter once MCP server is stable |
| Memory expiry | Retain forever vs TTL-based S3 lifecycle rule | No expiry initially; add 365-day lifecycle rule on `archived` tagged objects if storage grows |
| Cert CN assertion strictness | Exact CN match vs prefix match | Exact match (`mcp-client.brad-duhon.com`) — no flexibility needed for single client |
| MCP server hosting | Local workstation only vs also available on remote dev host | Local only initially; add remote host support when needed by referencing same Secrets Manager ARNs |
