# engram

A personal memory layer for Agentic AI that persists context across many AI interfaces. Secured at the transport layer and living entirely in your own infrastructure.

## Overview

Engram is a serverless AWS microservice that gives Claude persistent, semantically-searchable memory across all conversations. Vectors are generated by Amazon Bedrock Titan Embeddings v2 (1024 dimensions), stored in S3 Vector Tables, and accessed via a local MCP server over mutual TLS (mTLS).

See [CLAUDE.md](CLAUDE.md) for full architecture, conventions, and phase implementation order.

## Prerequisites

- AWS account with sufficient permissions (AdministratorAccess recommended for initial deployment)
- AWS CLI v2 with a configured profile
- Terraform >= 1.5
- A domain name with a DNS zone you control (any registrar or DNS provider)
- Python 3.12+

## Deployment

Phases must be deployed in order. Each phase depends on all prior phases.

### Phase 0: Bootstrap TF State Backend

Creates the S3 bucket and DynamoDB table used as the Terraform remote state backend. Uses local state and is run once.

```bash
cd terraform/bootstrap
terraform init
terraform apply
```

Then initialize the root module with the remote backend:

```bash
cd terraform
cp backend.hcl.example backend.hcl  # fill in bucket, region, profile
terraform init -backend-config=backend.hcl
```

Create a `terraform/terraform.tfvars` (gitignored) with your values:

```hcl
aws_region         = "us-east-1"
aws_profile        = "<your-aws-profile>"
server_domain_name = "memory.<your-domain>"
client_domain_name = "mcp-client.<your-domain>"
```

### Phase 1: Storage and Networking

Creates the VPC, private subnets, S3 Gateway Endpoint, security groups, S3 Vectors bucket, and vector index.

```bash
terraform apply -target=module.networking -target=module.storage
```

After apply, create the vector index (no Terraform resource exists yet):

```bash
./scripts/create-vector-index.sh
```

### Phase 2: Certificates

Creates two ACM certificates (server + exportable mTLS client) and Secrets Manager shells for the exported key material.

```bash
terraform apply -target=module.certificates.aws_acm_certificate.server \
                -target=module.certificates.aws_acm_certificate.client \
                -target=module.certificates.aws_secretsmanager_secret.client_cert \
                -target=module.certificates.aws_secretsmanager_secret.client_cert_passphrase
```

**Manual DNS validation step (required):**

After the certificates are created, ACM generates CNAME records that prove domain ownership. You must add these to your DNS zone before the certificates will reach `ISSUED` status. Retrieve the records:

```bash
terraform output -json
```

> **Note for deployers:** Engram does not assume your DNS zone is in AWS or in the same account. Add the CNAME records through whatever DNS provider or account controls your zone. The records are permanent -- they must remain in place for ACM automatic renewal to work.

Once the CNAMEs are live and propagated (typically 1-5 minutes), complete the Phase 2 apply:

```bash
terraform apply -target=module.certificates
```

After both certs are `ISSUED`, export the client certificate leaf cert and private key into Secrets Manager:

```bash
python scripts/export_client_cert.py
```

### Phase 3: Compute

Creates the memory handler Lambda, cert rotator Lambda, IAM roles, Bedrock Interface Endpoint, and S3 Vectors Interface Endpoint.

```bash
terraform apply -target=module.compute
```

Smoke test end-to-end (requires deployed infrastructure):

```bash
./scripts/smoke-test.sh
```

## Directory Structure

```
engram/
  CLAUDE.md                   # Architecture conventions for implementers
  terraform/
    bootstrap/                # One-time TF state backend setup (local state)
    modules/
      networking/             # VPC, subnets, endpoints, security groups
      storage/                # S3 Vectors bucket + index
      certificates/           # ACM certs, Secrets Manager shells
      compute/                # Lambda functions, IAM roles, VPC endpoints
    main.tf                   # Root module -- wires all phase modules
  docs/phases/                # Per-phase implementation specifications
  scripts/                    # Operational scripts (cert export, smoke test)
  src/
    memory_handler/           # Lambda -- store/recall/summarize via Bedrock + S3 Vectors
    cert_rotator/             # Lambda -- ACM cert re-export to Secrets Manager
  mcp_server/                 # Local MCP server (Phase 5+)
  hooks/                      # Claude Code hook scripts (Phase 7)
```

## Security Notes

- mTLS is enforced at API Gateway. The truststore contains only the leaf certificate of the specific ACM client certificate created in Phase 2 -- no other certificates are trusted.
- Lambda runs in a VPC with no internet access. S3 is accessed via Gateway Endpoint; Bedrock and S3 Vectors via Interface Endpoints with scoped endpoint policies.
- The client certificate private key is encrypted with a random passphrase in Secrets Manager. It never exists in Terraform state.
- Lambda dependencies (pydantic v2, Powertools) are supplied via the official AWS Lambda Powertools layer -- no build step required.
- The PostCompact hook (Phase 7) decrypts the private key via bash process substitution so plaintext never touches disk.
